# ShadowDistributedRouter 向量意图路由架构文档

Date: 2026-05-15  
Status: Architecture Blueprint / No Code Modification Yet  
Scope: 仅记录方案，不修改代码

---

## 1. 背景与目标

当前 `ShadowDistributedRouter` 已实现基础分布式同名插件路由能力：

- 多设备注册同名插件时，主注册表仍保持单一工具名；
- Shadow 插件维护影子注册表；
- 当某个 VChat 发起请求时，优先根据请求来源设备调用对应设备上的分布式插件；
- 若无法判断来源，则 fallback 到原有分布式路由策略。

现在需要增强能力：

> 用户可以用自然语言显式指定目标设备，例如：“调用家里电脑的文件管理插件，列一下 G 盘。”

目标是在 **不修改 VCP 核心源码** 的前提下，仅通过增强 `ShadowDistributedRouter` 插件，实现自然语言设备意图路由。

## 2. 设计约束

### 2.1 不修改核心源码

不得修改：

- `modules/chatCompletionHandler.js`
- `modules/vcpLoop/toolExecutor.js`
- `modules/handlers/streamHandler.js`
- `modules/handlers/nonStreamHandler.js`
- `Plugin/RAGDiaryPlugin/*`
- 其他 VCP 核心模块

允许修改：

- `Plugin/ShadowDistributedRouter/ShadowDistributedRouter.js`
- `Plugin/ShadowDistributedRouter/device-aliases.json`
- Shadow 插件目录下的文档、配置文件

### 2.2 不引入轻量大模型

不使用 Gemini Flash-Lite 或其他额外 LLM 做设备目标抽取。

原因：

- 增加外部依赖；
- 增加延迟；
- 增加成本；
- 路由决策变成不可审计的模型输出。

最终方案改为复用 VCP 现有向量化能力。

### 2.3 用户别名手动配置

设备别名配置文件中的 `friendlyNames` 必须由用户手动配置。

系统可以自动生成和同步：

- `serverId`
- `hostName`
- `serverName`
- `lastKnownIPs`
- `enabled`
- `updatedAt`
- `autoGenerated`

系统不得自动臆测或覆盖：

- `friendlyNames`
- `note`
- 用户手工设置的 `enabled`

## 3. 已确认源码事实

### 3.1 ToolExecutor 可被运行时 patch

源码确认：

```js
module.exports = ToolExecutor;
```

因此 Shadow 插件可在运行时：

```js
const ToolExecutor = require('../../modules/vcpLoop/toolExecutor');
```

并 patch：

```js
ToolExecutor.prototype.execute
```

若 `execute` 未被实例私有化绑定，patch 原型方法即可影响既有实例调用链。

### 3.2 ToolExecutor.execute 拥有上下文消息

源码确认其函数形态类似：

```js
async execute(toolCall, clientIp, contextMessages = []) {
  const { name, args, river, vref } = toolCall;
  const result = await this.pluginManager.processToolCall(name, args, clientIp, 'post');
}
```

其中：

- `toolCall`：当前工具调用；
- `clientIp`：请求来源 IP；
- `contextMessages`：当轮对话上下文。

所以，若只改 Shadow 插件，最佳拦截点是：

```js
ToolExecutor.prototype.execute()
```

### 3.3 VCP 已有消息文本提取逻辑

源码中已有类似函数：

```js
function getMessageTextContent(msg) {
  if (!msg) return '';
  if (typeof msg.content === 'string') return msg.content;
  if (Array.isArray(msg.content)) {
    return msg.content
      .filter(part => part.type === 'text')
      .map(part => part.text)
      .join('\n');
  }
  return '';
}
```

并已有排除系统消息 / 工具 payload 的真实用户消息提取逻辑：

```js
const lastUserIndex = contextMessages.findLastIndex(msg => {
  if (msg.role !== 'user') return false;
  const content = getMessageTextContent(msg);
  return !content.startsWith('<!-- VCP_TOOL_PAYLOAD -->') &&
    !content.startsWith('[系统提示:]') &&
    !content.startsWith('[系统邀请指令:]');
});
```

Shadow 可复制该逻辑到自身文件中，不修改原文件。

### 3.4 VCP 已有向量化与相似度能力

源码确认 `EmbeddingUtils.js` 导出：

```js
module.exports = { getEmbeddingsBatch, cosineSimilarity };
```

`ToolExecutor` 已使用：

```js
const { getEmbeddingsBatch, cosineSimilarity } = require('../../EmbeddingUtils');
```

同时，工具执行层已有使用 RAG 插件缓存向量的路线：

```js
const ragPlugin = this.pluginManager?.messagePreprocessors?.get('RAGDiaryPlugin');

if (ragPlugin && typeof ragPlugin.getBatchEmbeddingsCached === 'function') {
  const allVectors = await ragPlugin.getBatchEmbeddingsCached(allTexts);
}
```

因此 Shadow 向量路由的优先级为：

1. 优先使用 `RAGDiaryPlugin.getBatchEmbeddingsCached()`;
2. 失败时 fallback 到 `EmbeddingUtils.getEmbeddingsBatch()`;
3. 相似度计算使用 `EmbeddingUtils.cosineSimilarity()`。

## 4. 总体架构

新增能力仍然全部内聚在 Shadow 插件内：

```txt
ShadowDistributedRouter
├─ 原有：分布式同名插件影子注册表 shadowRegistry
├─ 原有：设备信息注册表 serverInfoRegistry
├─ 原有：requestIp / sourceNode 来源亲和路由
├─ 新增：device-aliases.json 设备别名配置
├─ 新增：自动同步设备系统字段
├─ 新增：设备描述向量索引 aliasVectorIndex
├─ 新增：patch ToolExecutor.prototype.execute
├─ 新增：从 contextMessages 提取最后真实用户消息
├─ 新增：用户消息向量化
├─ 新增：AsyncLocalStorage 传递 routeContext
└─ 新增：resolveTargetServerId 前置向量意图路由层
```

## 5. 完整运行链路

用户输入：

```txt
调用家里电脑的文件管理插件，列一下 G 盘。
```

模型生成工具调用：

```txt
FileOperator.ListDirectory({ directoryPath: "G:\\" })
```

运行链路：

```txt
ToolExecutor.execute(toolCall, clientIp, contextMessages)
        ↓
Shadow patch 拦截 execute()
        ↓
提取 contextMessages 中最后一条真实用户消息
        ↓
向量化 userText
        ↓
requestContext.run({ requestIp, sourceNode, routeContext }, ...)
        ↓
原始 ToolExecutor.execute() 继续执行
        ↓
pluginManager.processToolCall(...)
        ↓
Shadow patchProcessToolCall() 继承父级 AsyncLocalStorage store
        ↓
wss.executeDistributedTool(...)
        ↓
Shadow resolveTargetServerId()
        ↓
优先执行精确别名匹配 / 向量匹配
        ↓
命中目标设备 serverId
        ↓
确认该设备拥有当前 toolName 的 shadow instance
        ↓
路由到指定设备
```

若任意环节失败：

```txt
低置信度 / 匹配歧义 / 向量化失败 / 目标无工具实例
        ↓
fallback 到原有 requestIp/sourceNode 来源亲和逻辑
```

## 6. 设备配置文件设计

文件路径：

```txt
Plugin/ShadowDistributedRouter/device-aliases.json
```

建议结构：

```json
{
  "vectorRoute": {
    "enabled": true,
    "threshold": 0.72,
    "ambiguousMargin": 0.08,
    "maxUserTextLength": 1000,
    "preferRagCache": true,
    "fallbackToEmbeddingUtils": true,
    "debug": true
  },
  "devices": [
    {
      "serverId": "DESKTOP-P23SB1E",
      "friendlyNames": ["家里电脑", "书房主机", "主力机"],
      "hostName": "DESKTOP-P23SB1E",
      "serverName": "DESKTOP-P23SB1E",
      "lastKnownIPs": ["192.168.1.23", "100.100.50.10"],
      "username": "",
      "note": "墨行家里的书房 Windows 主机",
      "enabled": true,
      "autoGenerated": true,
      "updatedAt": "2026-05-15T00:00:00.000Z"
    }
  ]
}
```

### 字段说明

#### vectorRoute.enabled

是否启用向量意图路由。

关闭时完全回到原有 Shadow 路由逻辑。

#### vectorRoute.threshold

向量匹配最低相似度阈值。

建议初始值：

```txt
0.72
```

宁可 fallback，不可错路由。

#### vectorRoute.ambiguousMargin

Top1 与 Top2 的最低差值。

建议初始值：

```txt
0.08
```

若 Top1 和 Top2 太接近，判定为歧义，不进行显式路由。

#### friendlyNames

用户手动配置的设备别名。

例如：

```json
["家里电脑", "书房主机", "主力机"]
```

系统绝不覆盖。

#### serverId / hostName / serverName / lastKnownIPs

由 Shadow 根据注册表自动同步。

来源包括：

- `shadowRegistry`
- `serverInfoRegistry`
- 分布式插件注册信息

#### username

只有当客户端注册信息已经上报用户名时才可自动填写。

否则保持空字符串。

Shadow 不应凭空推断远端 Windows 用户名。

#### note

用户可手动填写。

用于增强设备描述文本，提高向量匹配质量。

## 7. 配置自动生成与合并策略

Shadow 启动或设备注册更新时执行同步：

1. 读取已有 `device-aliases.json`；
2. 从 `shadowRegistry` 和 `serverInfoRegistry` 收集当前已知设备；
3. 如果某个 `serverId` 不存在于配置文件，则自动新增设备条目；
4. 如果某个 `serverId` 已存在，只更新系统字段：
   - `hostName`
   - `serverName`
   - `lastKnownIPs`
   - `updatedAt`
   - `autoGenerated`
5. 不覆盖用户字段：
   - `friendlyNames`
   - `note`
   - `enabled`

原则：

```txt
机器字段自动同步。
人类命名手动维护。
```

## 8. 用户消息提取逻辑

Shadow 内部实现：

```js
function getMessageTextContent(msg) {
  if (!msg) return '';
  if (typeof msg.content === 'string') return msg.content;

  if (Array.isArray(msg.content)) {
    return msg.content
      .filter(part => part.type === 'text')
      .map(part => part.text)
      .join('\n');
  }

  return '';
}

function getLastRealUserText(contextMessages = []) {
  if (!Array.isArray(contextMessages)) return '';

  for (let i = contextMessages.length - 1; i >= 0; i--) {
    const msg = contextMessages[i];
    if (msg?.role !== 'user') continue;

    const content = getMessageTextContent(msg).trim();
    if (!content) continue;

    if (content.startsWith('<!-- VCP_TOOL_PAYLOAD -->')) continue;
    if (content.startsWith('[系统提示:]')) continue;
    if (content.startsWith('[系统邀请指令:]')) continue;

    return content;
  }

  return '';
}
```

## 9. 向量化能力探测

Shadow 内部实现统一向量化函数：

```js
async function embedTexts(texts, pluginManager) {
  const cleanTexts = texts
    .filter(Boolean)
    .map(t => String(t).slice(0, 1000));

  if (cleanTexts.length === 0) return [];

  const ragPlugin = pluginManager?.messagePreprocessors?.get?.('RAGDiaryPlugin');

  if (ragPlugin && typeof ragPlugin.getBatchEmbeddingsCached === 'function') {
    try {
      return await ragPlugin.getBatchEmbeddingsCached(cleanTexts);
    } catch (err) {
      console.warn('[ShadowDistributedRouter] RAG embedding cache failed:', err.message);
    }
  }

  const { getEmbeddingsBatch } = require('../../EmbeddingUtils');

  const embeddingConfig = {
    apiKey: process.env.API_KEY,
    apiUrl: process.env.API_URL,
    model: process.env.WhitelistEmbeddingModel || 'google/gemini-embedding-001'
  };

  return await getEmbeddingsBatch(cleanTexts, embeddingConfig);
}
```

说明：

- 优先复用 RAG 缓存；
- 缓存不可用时使用 VCP 原有 `EmbeddingUtils`;
- 该逻辑只位于 Shadow 插件内；
- 不修改 RAG 插件源码。

## 10. 设备别名向量索引

Shadow 加载配置后，为每台设备生成描述文本：

```txt
设备别名：家里电脑、书房主机、主力机。
主机名：DESKTOP-P23SB1E。
服务器名：DESKTOP-P23SB1E。
IP：192.168.1.23、100.100.50.10。
说明：墨行家里的书房 Windows 主机。
```

不建议只向量化短别名，如：

```txt
家里电脑
```

原因：

- 短词语义薄；
- 向量区分度低；
- 易与其他设备产生歧义。

建议索引结构：

```js
this.aliasVectorIndex = [
  {
    serverId,
    aliasText,
    vector,
    device
  }
];
```

## 11. patch ToolExecutor.prototype.execute

Shadow 初始化时执行：

```js
patchToolExecutorExecute() {
  const ToolExecutor = require('../../modules/vcpLoop/toolExecutor');

  if (!ToolExecutor?.prototype?.execute) {
    console.warn('[ShadowDistributedRouter] ToolExecutor.prototype.execute not found.');
    return false;
  }

  const original = ToolExecutor.prototype.execute;

  if (original.__shadowVectorRoutePatched) {
    return true;
  }

  const self = this;

  const patched = async function patchedExecute(toolCall, clientIp, contextMessages = [], ...rest) {
    let routeContext = null;

    try {
      const userText = self.getLastRealUserText(contextMessages);

      if (userText) {
        const vectors = await self.embedTexts([userText], this.pluginManager);
        const userVector = vectors?.[0] || null;

        routeContext = {
          userText,
          userVector,
          createdAt: Date.now()
        };
      }
    } catch (err) {
      console.warn('[ShadowDistributedRouter] build routeContext failed:', err.message);
    }

    const parentStore = requestContext.getStore() || {};

    return requestContext.run({
      ...parentStore,
      requestIp: clientIp || parentStore.requestIp,
      sourceNode: parentStore.sourceNode || 'post',
      routeContext
    }, async () => {
      return original.call(this, toolCall, clientIp, contextMessages, ...rest);
    });
  };

  patched.__shadowVectorRoutePatched = true;
  patched.__shadowOriginal = original;

  ToolExecutor.prototype.execute = patched;

  this.patchRecords.push({
    target: ToolExecutor.prototype,
    method: 'execute',
    original,
    patched
  });

  return true;
}
```

## 12. patchProcessToolCall 必须继承父级 store

若现有逻辑为：

```js
requestContext.run({ requestIp, sourceNode }, ...)
```

会覆盖 `ToolExecutor.execute()` 注入的 `routeContext`。

必须改为：

```js
const patched = async function patchedProcessToolCall(toolName, toolArgs, requestIp = null, sourceNode = null) {
  const parentStore = requestContext.getStore() || {};

  return requestContext.run({
    ...parentStore,
    requestIp: requestIp || parentStore.requestIp,
    sourceNode: sourceNode || parentStore.sourceNode
  }, async () => {
    return original.call(this, toolName, toolArgs, requestIp, sourceNode);
  });
};
```

这是方案成败关键。

## 13. 路由决策顺序

在 `resolveTargetServerId()` 原有逻辑之前插入：

```js
resolveTargetServerId(wss, originalServerId, toolName) {
  const store = requestContext.getStore();

  const vectorTarget = this.resolveVectorRouteTarget(toolName, store?.routeContext);
  if (vectorTarget) {
    return vectorTarget;
  }

  // 原有 requestIp / sourceNode 来源亲和逻辑
}
```

决策顺序：

1. 检查 `vectorRoute.enabled`;
2. 检查是否存在 `routeContext.userText`;
3. 检查当前 `toolName` 是否存在 shadow instances；
4. 精确别名匹配；
5. 向量相似度匹配；
6. 歧义拒绝；
7. 目标设备工具实例校验；
8. 命中则返回目标 `serverId`;
9. 失败则 fallback 到原有来源亲和逻辑。

## 14. 精确匹配逻辑

优先执行字符串包含匹配：

```js
function normalizeText(text) {
  return String(text || '')
    .trim()
    .toLowerCase()
    .replace(/\s+/g, '');
}

resolveExactAliasTarget(toolName, userText) {
  const instances = shadowRegistry.get(toolName);
  if (!instances) return null;

  const normalizedUserText = normalizeText(userText);

  for (const device of this.deviceAliases.devices) {
    if (!device.enabled) continue;
    if (!instances.has(device.serverId)) continue;

    for (const alias of device.friendlyNames || []) {
      if (alias && normalizedUserText.includes(normalizeText(alias))) {
        return device.serverId;
      }
    }
  }

  return null;
}
```

优点：

- 快；
- 稳；
- 不依赖向量；
- 用户明确说出别名时最可靠。

## 15. 向量匹配逻辑

向量匹配用于处理近义表达：

```txt
用家里那台主力机看一下 G 盘
让书房那台机器列目录
打开我家里的 Windows 主机文件管理
```

伪代码：

```js
resolveVectorRouteTarget(toolName, routeContext) {
  if (!this.vectorRouteConfig.enabled) return null;
  if (!routeContext?.userText) return null;

  const instances = shadowRegistry.get(toolName);
  if (!instances || instances.size === 0) return null;

  const exact = this.resolveExactAliasTarget(toolName, routeContext.userText);
  if (exact) return exact;

  if (!routeContext.userVector) return null;
  if (!Array.isArray(this.aliasVectorIndex) || this.aliasVectorIndex.length === 0) return null;

  const scored = this.aliasVectorIndex
    .filter(item => item.vector && instances.has(item.serverId))
    .map(item => ({
      ...item,
      score: cosineSimilarity(routeContext.userVector, item.vector)
    }))
    .sort((a, b) => b.score - a.score);

  const top1 = scored[0];
  const top2 = scored[1];

  if (!top1) return null;
  if (top1.score < this.vectorRouteConfig.threshold) return null;

  if (top2 && top1.score - top2.score < this.vectorRouteConfig.ambiguousMargin) {
    return null;
  }

  return top1.serverId;
}
```

## 19. 风险与护栏

### 19.1 patch 风险

`ToolExecutor.prototype.execute` 若未来改为实例绑定方法，原型 patch 可能失效。

护栏：

- 初始化时检查方法存在；
- 打印 patch 成功日志；
- Phase 0 探针验证。

### 19.2 routeContext 被覆盖

若 `patchProcessToolCall` 不继承父级 store，会丢失 `routeContext`。

护栏：

- 必须使用 `{ ...parentStore, requestIp, sourceNode }`；
- 不可直接 `requestContext.run({ requestIp, sourceNode })`。

### 19.3 向量误判

向量匹配可能把相近设备混淆。

护栏：

- 精确匹配优先；
- 阈值保守；
- Top1/Top2 差值不足则拒绝；
- 目标设备无工具则拒绝；
- 失败 fallback。

### 19.4 用户配置被覆盖

自动同步可能误伤用户手写别名。

护栏：

- 永不覆盖 `friendlyNames`;
- 永不覆盖 `note`;
- 谨慎处理 `enabled`;
- 只同步系统字段。

### 19.5 向量化失败

RAG 缓存或 EmbeddingUtils 可能失败。

护栏：

- catch 错误；
- debug 日志；
- fallback 到原有来源亲和；
- 不影响工具调用。

## 20. 最终定案

最终方案：

```txt
只修改 ShadowDistributedRouter 插件目录
        ↓
运行时 patch ToolExecutor.prototype.execute
        ↓
读取 contextMessages
        ↓
提取最后真实用户消息
        ↓
复用 VCP 现有向量化能力
        ↓
设备别名配置中 friendlyNames 由用户手动配置
        ↓
其他设备字段由 Shadow 自动从注册表同步
        ↓
构建设备描述向量索引
        ↓
resolveTargetServerId 前置执行精确别名 / 向量匹配
        ↓
匹配成功则路由到指定 serverId
        ↓
失败则 fallback 到原有 requestIp/sourceNode 来源亲和
```

该方案：

- 不修改核心源码；
- 不引入轻量模型；
- 不污染工具参数；
- 不覆盖用户命名；
- 不破坏现有路由；
- 可分阶段验证；
- 可回滚。
