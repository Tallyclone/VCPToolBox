# ShadowDistributedRouter 向量意图路由实现蓝图

日期：2026-05-15  
项目：VCPToolBox / Plugin / ShadowDistributedRouter  
目标：在不修改 VCP 核心源码的前提下，仅通过 ShadowDistributedRouter 插件实现“自然语言显式指定设备”的分布式插件路由能力。

---

## 1. 背景

当前 ShadowDistributedRouter 已实现：

> 哪个 VChat 发起请求，就优先调用哪个设备上的同名分布式插件。

该能力基于：

- `requestIp`
- `sourceNode`
- `shadowRegistry`
- `serverInfoRegistry`
- `AsyncLocalStorage requestContext`

现有策略属于“来源亲和路由”。

下一阶段目标是支持自然语言指定设备，例如：

- “调用家里电脑的文件管理插件”
- “让书房主机列一下 G 盘”
- “用 NAS 那台机器执行文件搜索”

要求：

1. 不修改 VCP 核心源码。
2. 不新增外部轻量模型依赖。
3. 复用 VCP 已有向量化能力。
4. 只修改 `Plugin/ShadowDistributedRouter/ShadowDistributedRouter.js`，以及新增 Shadow 插件自己的配置文件。
5. `friendlyNames` 由用户手动配置，系统不得覆盖。
6. 其他设备字段由 Shadow 自动从现有注册表同步生成。
7. 匹配失败、歧义、向量化失败时，必须 fallback 到现有来源亲和策略。

---

## 2. 核心方案

通过 ShadowDistributedRouter 运行时 patch 上游执行函数：

```text
ToolExecutor.prototype.execute(toolCall, clientIp, contextMessages)
```

在工具执行前，Shadow 从 `contextMessages` 中提取当轮最后一条真实用户消息，并复用 VCP 现有 embedding 管线生成用户消息向量。

随后通过 `AsyncLocalStorage` 将 `routeContext` 传递到 Shadow 路由决策层。

最终在 `resolveTargetServerId()` 中执行：

```text
精确 friendlyNames 字符串命中
  -> 向量相似度匹配
  -> 歧义拒绝
  -> 工具实例可用性校验
  -> 成功返回目标 serverId
  -> 失败 fallback 到原 requestIp/sourceNode 来源亲和逻辑
```

---

## 3. 重要架构决策

### 3.1 只改 Shadow 插件，不改核心源码

允许 Shadow 在运行时 patch：

- `ToolExecutor.prototype.execute`
- `pluginManager.processToolCall`
- `wss.executeDistributedTool`

但不直接修改这些源码文件。

这是 Shadow 插件本身的定位：运行时路由补丁层。

---

### 3.2 routeContext 使用 AsyncLocalStorage 传递

主路径不使用 Symbol 挂载到 `toolArgs`。

原因：

1. 当前方案已 patch 到 `ToolExecutor.execute()` 层，可以直接开启 ALS 上下文。
2. 使用 ALS 不污染工具参数。
3. 避免真实插件收到非预期字段。
4. `patchProcessToolCall()` 只需继承父级 store，routeContext 即可一路传到 `resolveTargetServerId()`。

链路：

```text
patch ToolExecutor.prototype.execute()
  -> requestContext.run({ requestIp, sourceNode, routeContext }, ...)
  -> original execute()
  -> pluginManager.processToolCall()
  -> Shadow.patchProcessToolCall() 继承 parentStore
  -> wss.executeDistributedTool()
  -> Shadow.resolveTargetServerId()
  -> requestContext.getStore().routeContext
```

---

### 3.3 friendlyNames 用户手动配置

设备配置中：

- `friendlyNames`：用户手动维护，系统不得覆盖。
- `note`：用户可选维护，系统不得覆盖。
- `enabled`：用户可手动控制，系统不得覆盖。

系统自动同步：

- `serverId`
- `hostName`
- `serverName`
- `lastKnownIPs`
- `updatedAt`
- `autoGenerated`

注意：

`username` 只有在客户端注册信息已经上报时才能自动生成。否则默认为空字符串，不可臆造。

## 5. 配置自动生成与合并策略

Shadow 启动或设备注册信息变化时，执行同步：

1. 读取现有 `device-aliases.json`。
2. 从 `serverInfoRegistry` 与 `shadowRegistry` 收集当前已知设备。
3. 如果某个 `serverId` 不存在于配置文件，则自动新增设备项。
4. 如果已存在，只更新系统字段：
   - `hostName`
   - `serverName`
   - `lastKnownIPs`
   - `updatedAt`
   - `autoGenerated`
5. 严禁覆盖用户字段：
   - `friendlyNames`
   - `note`
   - `enabled`
   - `username`，除非未来确认客户端稳定上报 username，且用户未手动填写。

推荐合并原则：

```text
用户字段优先。
系统字段同步。
缺失设备新增。
失联设备保留。
```

失联设备不应自动删除。最多标记：

```json
"lastSeenAt": "...",
"online": false
```

是否删除由用户决定。

## 6. 当轮真实用户消息提取

Shadow 通过 patch `ToolExecutor.prototype.execute()` 拿到第三参数：

```js
contextMessages
```

在 Shadow 内部复制轻量提取函数：

```js
function getMessageTextContent(msg) {
  const content = msg?.content;
  if (!content) return "";

  if (typeof content === "string") return content;

  if (Array.isArray(content)) {
    return content
      .map(part => {
        if (typeof part === "string") return part;
        if (part?.type === "text") return part.text || "";
        if (part?.text) return part.text;
        return "";
      })
      .filter(Boolean)
      .join("\n");
  }

  return "";
}

function getLastRealUserText(messages = []) {
  if (!Array.isArray(messages)) return "";

  for (let i = messages.length - 1; i >= 0; i--) {
    const msg = messages[i];
    if (msg?.role !== "user") continue;

    const text = getMessageTextContent(msg).trim();
    if (!text) continue;

    if (text.startsWith("<!-- VCP_TOOL_PAYLOAD -->")) continue;
    if (text.startsWith("[系统提示:]")) continue;
    if (text.startsWith("[系统邀请指令:]")) continue;

    return text;
  }

  return "";
}
```

## 7. 向量化能力复用

Shadow 不引入轻量模型。

优先级：

1. `RAGDiaryPlugin.getBatchEmbeddingsCached()`
2. `RAGDiaryPlugin.getSingleEmbeddingCached()`
3. `EmbeddingUtils.getEmbeddingsBatch()`

能力探测方式：

```js
function getRagPlugin(pluginManager) {
  return pluginManager?.messagePreprocessors?.get?.("RAGDiaryPlugin")
    || pluginManager?.plugins?.get?.("RAGDiaryPlugin")
    || null;
}
```

向量化函数建议：

```js
async function embedTexts(texts, pluginManager, config) {
  const cleanTexts = texts
    .filter(Boolean)
    .map(t => String(t).slice(0, config?.vectorRoute?.maxUserTextLength || 1000));

  if (cleanTexts.length === 0) return [];

  const ragPlugin = getRagPlugin(pluginManager);

  if (ragPlugin) {
    let processedTexts = cleanTexts;

    if (typeof ragPlugin.sanitizeForEmbedding === "function") {
      processedTexts = cleanTexts
        .map(t => ragPlugin.sanitizeForEmbedding(t, "user"))
        .filter(Boolean);
    }

    if (typeof ragPlugin.getBatchEmbeddingsCached === "function") {
      try {
        return await ragPlugin.getBatchEmbeddingsCached(processedTexts);
      } catch (err) {
        console.warn("[ShadowDistributedRouter] RAG batch embedding failed:", err.message);
      }
    }

    if (typeof ragPlugin.getSingleEmbeddingCached === "function") {
      try {
        return await Promise.all(
          processedTexts.map(t => ragPlugin.getSingleEmbeddingCached(t))
        );
      } catch (err) {
        console.warn("[ShadowDistributedRouter] RAG single embedding failed:", err.message);
      }
    }
  }

  if (config?.vectorRoute?.fallbackToEmbeddingUtils !== false) {
    const { getEmbeddingsBatch } = require("../../EmbeddingUtils");

    return await getEmbeddingsBatch(cleanTexts, {
      apiKey: process.env.API_KEY,
      apiUrl: process.env.API_URL,
      model: process.env.WhitelistEmbeddingModel || "google/gemini-embedding-001"
    });
  }

  return [];
}
```

完整版：

```js
patchToolExecutorExecute() {
  const ToolExecutor = require("../../modules/vcpLoop/toolExecutor");

  if (!ToolExecutor?.prototype?.execute) {
    console.warn("[ShadowDistributedRouter] ToolExecutor.prototype.execute not found.");
    return false;
  }

  const original = ToolExecutor.prototype.execute;
  const self = this;

  ToolExecutor.prototype.execute = async function patchedExecute(toolCall, clientIp, contextMessages = [], ...rest) {
    let routeContext = null;

    try {
      const userText = self.getLastRealUserText(contextMessages);

      if (userText) {
        const vectors = await self.embedTexts([userText], this.pluginManager || self.pluginManager, self.deviceAliasConfig);

        routeContext = {
          userText,
          userVector: vectors?.[0] || null,
          createdAt: Date.now()
        };
      }
    } catch (err) {
      console.warn("[ShadowDistributedRouter] build routeContext failed:", err.message);
    }

    const parentStore = requestContext.getStore() || {};

    return requestContext.run({
      ...parentStore,
      requestIp: clientIp || parentStore.requestIp,
      sourceNode: parentStore.sourceNode || "post",
      routeContext
    }, async () => {
      return original.call(this, toolCall, clientIp, contextMessages, ...rest);
    });
  };

  return true;
}
```

注意：

如果项目已有 patch 标记工具，应沿用，避免重复 patch。

## 9. patchProcessToolCall 必须继承父级 store

原逻辑若是：

```js
requestContext.run({ requestIp, sourceNode }, ...)
```

会覆盖 ToolExecutor 层写入的 `routeContext`。

必须改成继承：

```js
const patched = async function patchedProcessToolCall(toolName, toolArgs, requestIp = null, sourceNode = null) {
  const parentStore = requestContext.getStore() || {};

  return requestContext.run({
    ...parentStore,
    requestIp: requestIp || parentStore.requestIp,
    sourceNode: sourceNode || parentStore.sourceNode
  }, async () => {
    return original.call(this, toolName, toolArgs, requestIp, sourceNode);
  });
};
```

## 10. 设备别名向量索引

生成设备描述文本时，不要只使用短别名。

推荐：

```js
function buildDeviceSemanticText(device) {
  const names = Array.isArray(device.friendlyNames)
    ? device.friendlyNames.filter(Boolean).join("、")
    : "";

  return [
    names ? `设备别名：${names}。` : "",

```

索引结构：


## 11. 路由决策流程

在 `resolveTargetServerId(wss, originalServerId, toolName)` 的来源亲和逻辑前增加：

```js
const store = requestContext.getStore();
const vectorTarget = this.resolveVectorRouteTarget(toolName, store?.routeContext);
if (vectorTarget) {
  return vectorTarget;
}
```

`resolveVectorRouteTarget()` 逻辑：

1. 检查 `vectorRoute.enabled`。
2. 检查 `routeContext.userText`。
3. 检查目标工具 `toolName` 是否存在 shadow instances。
4. 精确字符串匹配：
   - 用户文本包含某个 `friendlyNames`
   - 且该设备拥有当前 `toolName`
   - 直接返回该设备 `serverId`
5. 向量匹配：
   - `routeContext.userVector` vs `aliasVectorIndex.vector`
   - 按余弦相似度降序排序
6. 阈值判断：
   - Top1 score >= `threshold`
7. 歧义拒绝：
   - Top1 - Top2 >= `ambiguousMargin`
8. 工具可用性校验：
   - `shadowRegistry.get(toolName).has(serverId)`
9. 成功返回 `serverId`
10. 失败返回 `null`，继续原逻辑。

示意代码：

```js
resolveVectorRouteTarget(toolName, routeContext) {
  const cfg = this.deviceAliasConfig?.vectorRoute || {};
  if (!cfg.enabled) return null;
  if (!routeContext?.userText) return null;

  const instances = shadowRegistry.get(toolName);
  if (!instances || instances.size === 0) return null;

  const userText = normalizeText(routeContext.userText);

  // 1. 精确 friendlyNames 命中
  for (const device of this.deviceAliasConfig.devices || []) {
    if (!device.enabled) continue;
    if (!instances.has(device.serverId)) continue;

    for (const alias of device.friendlyNames || []) {
      if (!alias) continue;
      if (userText.includes(normalizeText(alias))) {
        return device.serverId;
      }
    }
  }

  // 2. 向量匹配
  if (!routeContext.userVector) return null;
  if (!Array.isArray(this.aliasVectorIndex) || this.aliasVectorIndex.length === 0) return null;

  const { cosineSimilarity } = require("../../EmbeddingUtils");

  const scored = this.aliasVectorIndex
    .filter(item => item.vector && instances.has(item.serverId))
    .map(item => ({
      ...item,
      score: cosineSimilarity(routeContext.userVector, item.vector)
    }))
    .sort((a, b) => b.score - a.score);

  const top1 = scored[0];
  const top2 = scored[1];

  const threshold = cfg.threshold ?? 0.72;
  const ambiguousMargin = cfg.ambiguousMargin ?? 0.08;

  if (!top1) return null;
  if (top1.score < threshold) return null;
  if (top2 && top1.score - top2.score < ambiguousMargin) return null;

  return top1.serverId;
}
```

## 12. normalizeText

用于精确匹配前的轻量归一：

```js
function normalizeText(text) {
  return String(text || "")
    .trim()
    .toLowerCase()
    .replace(/\s+/g, "");
}
```

后续可扩展：

- 全角半角归一
- 中文标点剔除
- 常见同义短语规则

第一版保持克制。

## 14. 风险与护栏

### 14.1 patch 重复执行

必须给 patched 函数打标记，防止重复 patch。

建议：

```js
const PATCH_FLAG = Symbol.for("ShadowDistributedRouter.ToolExecutorPatched");
if (ToolExecutor.prototype.execute[PATCH_FLAG]) return true;
patched[PATCH_FLAG] = true;
patched.__shadowOriginal = original;
```

---

### 14.2 ALS 上下文被覆盖

`patchProcessToolCall()` 必须继承 parent store。

这是核心护栏。

---

### 14.3 向量化延迟

用户文本向量化发生在工具执行前，会增加少量延迟。

缓解：

1. 优先使用 RAG 缓存。
2. 可增加本地短期 cache：
   - key: userText hash
   - value: vector
   - TTL: 60 秒
3. 设备别名向量启动时预构建，不在每次路由时重复生成。

---

### 14.4 整句用户文本语义被任务稀释

例如：

> 调用家里电脑的文件管理插件，列一下 G 盘最近修改的文件

整句包含设备、工具、操作三类语义。

护栏：

1. 先做 friendlyNames 精确匹配。
2. 设备 aliasText 使用完整描述句，而非短词。
3. 阈值保守，不强行命中。
4. 歧义时 fallback。

---

### 14.5 用户配置被覆盖

配置同步必须遵守：

```text
friendlyNames / note / enabled / username：用户优先
serverId / hostName / serverName / lastKnownIPs / updatedAt：系统同步
```

---

## 15. 最终结论

该方案成立。

它满足：

- 不修改 VCP 核心源码。
- 不引入外部轻量模型。
- 复用现有向量化系统。
- Shadow 插件自洽完成上下文获取、向量化、设备语义匹配、路由决策。
- 匹配失败时无损回退到现有来源亲和策略。

最终路线：

```text
Shadow patch ToolExecutor.prototype.execute
  -> 提取当轮真实用户消息
  -> 复用 VCP embedding 管线生成用户向量
  -> AsyncLocalStorage 传递 routeContext
  -> resolveTargetServerId 优先执行别名/向量路由
  -> 成功定向设备
  -> 失败 fallback 原逻辑
```

第一步必须先做探针验证。  
探针通过后，再进入完整实现。
